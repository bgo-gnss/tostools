# Destination: `tos audit missing-attributes` — flag entities lacking GPS-required attributes

## Problem

REYK exposed the gap: `tosGPS timespan REYK` reports `✗ no TOS start` because
the station entity is missing its `date_start` attribute (compare HAUC: 15
attributes on station entity; REYK: 7). The
attribute-dates audit (Layers 1-5) flagged 20 misdated `serial_number` periods
on REYK's devices, all fixed — but the station-level **missing** `date_start`
is invisible to that audit because rule 3 only checks the dates of attributes
that *do* exist.

The catalog at `data/attribute_codes.yaml` already declares
`gps_required_for: [...]` per code — we have all the elements in place to
take the next step: an audit that walks an entity and flags every required
attribute it's missing.

Scope expansion the operator confirmed: this should generalize across the
three TOS entity classes — **stations, devices, infrastructure (monument)**.
The catalog's three top-level scopes (`devices:`, `locations:`, `stations:`)
already mirror this distinction. Walking all three from the start avoids a
second pass later.

## In scope

- New CLI verb `tos audit missing-attributes <STA>` — per-station walk that
  audits (a) the station entity itself, (b) each child device joined to it,
  (c) the monument(s) flagged as infrastructure.
- **Rule:** for each entity in scope, for each catalog code where the entity's
  subtype ∈ `gps_required_for`, flag when the entity has no open attribute
  period for that code.
- Reuse the Layers 2-5 infrastructure: station walker (`_resolve_station_entity`),
  suppression file machinery, triage emission scaffolding,
  `_parse_action_file` extensibility.
- New apply verb `add-attribute <id_entity> <code> <value> <date_from>` for
  `tos audit apply`. Triage pre-fills `<value>` from the catalog's
  `default_value` when present (e.g. `infrastructure_subtype` → `"GPS
  undirstaða"`); otherwise emits `<FILL_VALUE>` as a placeholder the
  operator must replace before applying.
- Tests mirroring the Layers 2-5 style — mocked TOSClient, per-rule unit
  coverage plus end-to-end happy-path against synthetic catalog fixtures.

## Out of scope (v1)

- Suggesting values for codes without a catalog `default_value` — the audit
  flags the gap; the operator decides what value goes in.
- Auto-applying. Same triage workflow as attribute-dates: detect → review
  → apply.
- Fleet-wide sweep (`--all`). Per-station first; fleet later once the
  per-station verb has proven out.
- Fixing REYK as the v1 worked example. REYK's monument was incorrectly
  split into multiple monument records when it should be one (the per-survey
  changes belong on antenna offsets, not on new monument entities). Picking
  REYK as the demo means the fix path requires simultaneous coordinated
  edits across monument and antenna — too entangled for the v1 DoD. Use a
  cleanly-structured station (HAUC is a candidate) as the worked example.
- Layer 6.5 (separate work): an audit that flags
  `monument-with-multiple-records` as a structural error in TOS.

## Definition of done (4 layers, mirroring Layers 2-5)

1. **Catalog convention sweep** complete: `data/attribute_codes.yaml` `stations:`
   scope uses `geophysical` (the actual `code_entity_subtype`) instead of
   `jarðeðlisstöð` (Icelandic label). Same for `applies_to: [station]` →
   `applies_to: [geophysical]`. This is a prereq (project-todo in the hub);
   without it, the station-entity audit can't match catalog rules. 70
   occurrences across 43 entries.
2. `tos audit missing-attributes HAUC` walks the station + its devices +
   monument, flags every code where `gps_required_for` says "required" and
   the entity has no open period. Exit code 1 when anything's flagged, 0
   when clean.
3. `data/audit_suppressions/missing_attributes.txt` is read at audit time;
   adding a `SUPPRESS <id> <code>` line removes that entry from the next
   run. (Note: 2-tuple key, not the 3-tuple `(id, code, date_from)` used
   by the attribute-dates audit, since "missing" has no date_from to
   anchor on.)
4. End-to-end: `--triage` emits draft ACTION file → operator fills in
   values for codes without catalog defaults + uncomments → `tos audit
   apply --apply` → TOS reflects the new attribute values → re-running
   the verb shows the audited entity clean.

## Hard constraints

- Don't regress the 484/2 test suite.
- Reuse `audit_attribute_dates.load_catalog()` — same YAML, same scopes.
- Audit's view of "subtype" must match what TOS returns in
  `code_entity_subtype` (station: `geophysical`; devices: `gnss_receiver` /
  `antenna` / `radome` / `monument`). The catalog sweep (DoD step 1) is
  what aligns this.
- Triage file format must be parseable by the existing `_parse_action_file`
  — add `add-attribute` to `_SUPPORTED_VERBS` rather than building a
  parallel pipeline.

## Three-entity walking strategy

The walker fans out from the station entity:

```
station <STA>  (entity subtype=geophysical)
    │
    ├─ audit station-level attributes
    │       gps_required_for ⊇ geophysical → flag missing
    │       (catalog 'stations:' scope)
    │
    └─ for each open child join (children_connections with time_to=None):
        │
        ├─ device <subtype ∈ {gnss_receiver, antenna, radome, monument}>
        │       audit device-level attributes
        │       gps_required_for ⊇ <subtype> → flag missing
        │       (catalog 'devices:' scope)
        │
        └─ if subtype == 'monument':
                additionally walk infrastructure-only codes
                (catalog entries with monument-specific defaults like
                infrastructure_subtype, infrastructure_type, inscription)
```

For each entity, the rule:
```
for code, entry in catalog.items():
    if entity.code_entity_subtype not in entry['gps_required_for']:
        continue
    if entry.get('gps_relevance') != 'yes':
        continue
    if open_attribute_value(entity, code) is None:
        flag (entity_id, code, suggested_value=entry.get('default_value'))
```

## Interrogation history (compact)

Not run as a formal `/interrogate` session — the design crystallized during
the live REYK debugging exchange where the operator made the
generalization-to-three-entities call and surfaced the catalog convention
issue. Key alignment moments:

- Layer 6 is the natural Layer 6 of the attribute-dates audit work, not a
  separate project — same catalog, same machinery, complementary rule.
- Three entity scopes (station, devices, infrastructure) from the start —
  half-measures will require a re-walk.
- Catalog convention sweep is a hard prereq, not a side effect.
- REYK is **not** the worked example: monument fragmentation makes it an
  entanglement problem rather than a clean detection demo.
- Apply verb is `add-attribute`, not `set-attribute` — emphasizes the
  "missing → present" semantic and parallels `patch-attribute-date`'s naming.
